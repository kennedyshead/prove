# Prove v1.2 — Language Server

## Goal

Port `lsp.py` from the Python bootstrap compiler to the self-hosted Prove
compiler. After this release, `prove lsp` runs as a native LSP server
without Python or pygls.

**Depends on:** v1.1 (formatter, needed for code actions).

---

## Command

### `prove lsp`

- Start Language Server Protocol server on stdio
- Provides: diagnostics, go-to-definition, hover, code actions (suggestions)
- Communicates via JSON-RPC 2.0 over stdin/stdout

---

## Implementation

### LSP Protocol Handler

The Python version uses `pygls` to handle LSP protocol details. The
self-hosted version implements the protocol directly — no external
dependencies.

**JSON-RPC layer:**
- Read `Content-Length` header from stdin
- Parse JSON body using `Parse` module
- Dispatch to handler based on `method` field
- Serialize response as JSON, write with `Content-Length` header to stdout

**Stdlib dependencies:** `Parse` (JSON encode/decode), `InputOutput` (stdio),
`Text` (string building), `Table` (document cache).

### LSP Methods

| Method | Handler | Description |
|--------|---------|-------------|
| `initialize` | Return capabilities | Server metadata and feature flags |
| `initialized` | No-op | Client acknowledgment |
| `textDocument/didOpen` | Cache document, run diagnostics | New file opened |
| `textDocument/didChange` | Update cache, re-run diagnostics | File edited |
| `textDocument/didClose` | Remove from cache | File closed |
| `textDocument/diagnostic` | Lex → parse → check, return diagnostics | Pull diagnostics |
| `textDocument/hover` | Look up symbol under cursor | Type info and docs |
| `textDocument/definition` | Find declaration of symbol | Go-to-definition |
| `textDocument/codeAction` | Generate fix suggestions | Quick fixes from diagnostics |
| `shutdown` | Prepare to exit | Server shutdown request |
| `exit` | Terminate process | Final exit notification |

### Diagnostic Publishing

On every `didOpen` and `didChange`:
1. Lex the document text
2. Parse the token stream
3. Check the AST (semantic analysis)
4. Convert `Diagnostic` list to LSP diagnostic format
5. Publish via `textDocument/publishDiagnostics` notification

### Hover Information

When the cursor is over:
- **A function name:** Show verb, signature, return type, and doc comment
- **A type name:** Show type definition (record fields or algebraic variants)
- **A variable:** Show inferred type
- **A keyword:** Show brief description

### Code Actions

Map compiler `Suggestion` values to LSP `CodeAction`:
- Each suggestion becomes a `TextEdit` in a workspace edit
- Actions are linked to their originating diagnostic

### Document Cache

Maintain a `Table<DocumentState>` mapping URI → current text + last check
results. Re-check on every change (incremental checking is a future
optimization).

## Architecture

```
stdin → JSON-RPC reader → method dispatch → handler
                                              ↓
                                        compiler pipeline
                                        (lex → parse → check)
                                              ↓
                                        response builder
                                              ↓
stdout ← JSON-RPC writer ← response JSON ←───┘
```

The LSP server runs a single-threaded event loop reading from stdin. Each
request is processed synchronously. This is sufficient for single-file
editing; multi-file workspaces are a future enhancement.

## Parity Confirmation

LSP behavior must match between the Python and self-hosted versions.

### Automated Parity Tests

1. **Diagnostic parity:** Send `initialize` + `textDocument/didOpen` +
   `textDocument/diagnostic` to both servers for 10 test files. Compare
   the diagnostic arrays (code, severity, range, message).

2. **Hover parity:** For each test file, request hover at 5 symbol positions.
   Compare the hover content from both servers.

3. **Code action parity:** Open a file with known suggestions (e.g., a file
   triggering W310 implicit typing). Compare the code actions returned.

### Test Harness

Write a test script that:
1. Starts both LSP servers as subprocesses
2. Sends identical JSON-RPC messages to both
3. Collects responses
4. Compares response payloads (ignoring server metadata differences)

### Manual Spot Checks

- Open same file in editor with Python LSP, note diagnostics and hover info
- Switch to native LSP, verify identical behavior
- Trigger a code action, verify the edit is correct
- Verify server shuts down cleanly on `shutdown` + `exit`
