# Prove v0.8.2 ‚Äî proof ‚Üí explain Migration ‚úì

**Status: COMPLETE** ‚Äî 590 tests passing.

Merged the `proof` block into the `explain` block. The `explain` block now
supports both prose documentation and named entries with `when` conditions
for branching.

### What changed

| Before | After |
|--------|-------|
| `ProofBlock` | `ExplainBlock` |
| `ProofObligation` | `ExplainEntry` (name is now `str \| None`) |
| `FunctionDef.proof` + `FunctionDef.explain` | `FunctionDef.explain: ExplainBlock \| None` |
| `proof` keyword | removed ‚Äî use `explain` |
| `PROOF` token | removed |

### Syntax

Prose entries and named entries with `when` conditions coexist in one block:

```prove
transforms abs(n Integer) Integer
  ensures result >= 0
  explain
      return n for non - negative , negate for negative
      positive: identity when n >= 0
      negative: deducted when n < 0
from
    n
    0 - n
```

Named entries with `when` generate if/else-if chains in C. Prose entries
(no `name:` prefix) are documentation only.

### Files changed

| File | Change |
|------|--------|
| `ast_nodes.py` | `ProofBlock` ‚Üí `ExplainBlock`, `ProofObligation` ‚Üí `ExplainEntry`, merged FunctionDef fields |

| `parser.py` | `_parse_proof_block` ‚Üí `_parse_explain_block`, `_parse_proof_obligation` ‚Üí `_parse_explain_entry` + `_parse_named_explain_entry`, merged PROOF+EXPLAIN branches |

## Phase 1: Remaining Lint Codes

### 1a. E316 ‚Äî Name shadows builtin function

**File: `src/prove/checker.py`**

When a user-defined function or parameter has the same name as a builtin
function (`len`, `map`, `each`, `filter`, `reduce`, `to_string`, `clamp`,
`println`, `print`, `readln`), emit an error.

| Code | Message |
|------|---------|
| E316 | "'{name}' shadows the built-in function '{name}'. Choose a different name." |

**Where to check:**
- `_register_function()` (Pass 1) ‚Äî compare function name against builtin list
- `_check_function()` (Pass 2) ‚Äî compare parameter names against builtin list

**Implementation:**
- Add `_BUILTIN_FUNCTIONS: set[str]` at module level, populated from the names
  registered in `_register_builtins()`
- In `_register_function()`, if `fd.name in _BUILTIN_FUNCTIONS`, emit E316
- In parameter loop, if `p.name in _BUILTIN_FUNCTIONS`, emit E316

### 1b. E317 ‚Äî Name shadows builtin type

**File: `src/prove/checker.py`**

When a user-defined type has the same name as a builtin type, emit an error.

| Code | Message |
|------|---------|
| E317 | "'{name}' conflicts with the built-in type '{name}'. Choose a different name." |

**Where to check:**
- `_register_type()` (Pass 1) ‚Äî compare type name against `BUILTINS` dict from
  `types.py` (Integer, Decimal, Float, Boolean, String, Character, Byte, Unit)
  plus generic builtins (List, Option, Result, Table)

### 1c. W302 ‚Äî Unused import

**File: `src/prove/checker.py`**

When a name imported via `from Module use verb name` is never referenced in the
module body, emit a warning.

| Code | Message |
|------|---------|
| W302 | "'{name}' is imported from '{module}' but never used." |

**Implementation:**
- Add `self._used_imports: set[tuple[str, str]]` (module, name) to checker init
- When resolving a qualified call `Module.name(...)`, add `(module, name)` to set
- After Pass 2 completes, iterate `self._module_imports` and emit W302 for any
  `(module, name)` pair not in `_used_imports`
- Skip modules imported with bare `use Module` (all names are implicitly available)
- Stdlib modules used only for their type (e.g., `Table<V>` in a signature) count
  as used ‚Äî check type references too

### 1d. W303 ‚Äî Unused type definition

**File: `src/prove/checker.py`**

When a user-defined type is never referenced (not used in any parameter type,
return type, variable type, `satisfies`, or `match` pattern), emit a warning.

| Code | Message |
|------|---------|
| W303 | "Type '{name}' is defined but never used." |

**Implementation:**
- Add `self._used_types: set[str]` to checker init
- When resolving a `TypeExpr` (in `_resolve_type()`), add the type name to set
- After Pass 2 completes, iterate registered type definitions and emit W303 for
  any user-defined type not in `_used_types`
- Skip types exported via module interface (all top-level types are potentially
  exported ‚Äî only warn for types in non-library modules, or defer this to a
  future "unused export" lint)

### 1e. W323 ‚Äî ensures without explain

**File: `src/prove/prover.py`**

The `contracts.md` docs say: "`explain` is LSP-suggested, not compiler-required.
However, `ensures` without `explain` produces a **warning**."

Currently E390 fires as an **error** for this condition. This is too strict ‚Äî
`explain` is encouraged but should not block compilation.

| Code | Message |
|------|---------|
| W323 | "Function '{name}' has `ensures` but no `explain`. Document how each step satisfies the contract." |

**Migration from E390:**
- Change `_check_ensures_explain()` to emit **W323** (warning) instead of **E390** (error)
- Remove E390 from `DIAGNOSTIC_DOCS` registry
- Remove E390 section from `docs/diagnostics.md`
- Update `diagnostics.md` W323 entry to be the primary documentation
- In strict mode, W323 promotes to error automatically (existing behavior)

### 1f. W326 ‚Äî Recursion depth may be unbounded

**File: `src/prove/prover.py`**

When a recursive function has `terminates` but the measure expression is a
simple parameter (e.g., `terminates: n`), warn that the recursion depth may
be O(n) which could overflow the stack for large inputs.

| Code | Message |
|------|---------|
| W326 | "Recursive function '{name}' may have O(n) call depth. Consider an iterative approach or a logarithmic reduction." |

**Implementation:**
- Add `_check_recursion_depth()` to `ProofVerifier.verify()`
- If `fd.terminates is not None` and `self._calls_self(fd.name, fd.body)`:
  - If the terminates expression is a simple `IdentifierExpr` (just a parameter
    name), emit W326
  - If it's a more complex expression (e.g., `len(xs)`), still emit W326 since
    we can't statically prove the depth is small
  - If a `believe` references recursion bounds, suppress the warning
- This is a heuristic ‚Äî false positives are acceptable at warning level

## Phase 2: Lint Philosophy Documentation

### 2a. Create `docs/diagnostics-philosophy.md`

This page documents the fundamental principles behind Prove's diagnostic system.
It is central to the language identity ‚Äî Prove is a human-friendly language and
diagnostics are part of that promise.

**Content outline:**

```markdown
# Diagnostic Philosophy

## Why This Matters

Prove is designed to be read and understood by humans. Error messages are not
afterthoughts ‚Äî they are part of the language's user interface. Every diagnostic
follows these rules.

## Rule 1: Natural English

Messages are complete sentences. They start with a capital letter, end with a
period, and read like something a colleague would say.

    Bad:  "undefined type 'Foo'"
    Good: "Type 'Foo' is not defined."

## Rule 2: Show the Fix

Where possible, include a code suggestion or tell the user exactly what to do.
Don't just say what's wrong ‚Äî say how to fix it.

    "Cannot use '!' in 'parse' because it is not declared as failable.
     Add '!' after the return type."

## Rule 3: One Error at a Time

When the parser encounters a catastrophic failure, report only the first error.
Cascading errors from a single root cause are noise. Fix the first problem,
re-compile, and the cascading errors disappear.

## Rule 4: Three Severity Tiers

| Tier | Meaning | Action |
|------|---------|--------|
| **Error** | Won't compile | Must be fixed by hand |
| **Warning** | Compiles but should be improved | Should be fixed by hand |
| **Info** | Compiles and `prove format` can fix it | Run `prove format` |

Strict mode (`--strict`) promotes warnings to errors. Info stays info.

## Rule 5: Every Code Has Documentation

Every diagnostic code links to `prove.botwork.se/diagnostics/#CODE` with a
full explanation, before/after examples, and fix guidance. Codes are clickable
in editors via LSP.

## Rule 6: Errors for Broken, Warnings for Improvable

If the code compiles and runs correctly, it is not an error. Errors are reserved
for code that genuinely cannot be compiled. Style issues, missing documentation,
and code quality improvements are warnings or info.

## Rule 7: Suggestions Are Concrete

The `Suggestion` system provides machine-applicable fixes. When a diagnostic
includes a suggestion, the LSP can offer a one-click fix. Suggestions must be
syntactically valid Prove code.
```

## Future: Flat Multi-Arm Boolean Guards

The `explain` block with `when` conditions currently provides flat N-way boolean
dispatch (generating if/else-if chains in C). This is Prove's only mechanism for
multi-arm boolean branching ‚Äî `match` on a boolean only yields 2 arms (true/false).

A future version should design a proper language construct for this. Options:

1. **Guard syntax on match arms** ‚Äî like Rust/Haskell `match` with `if` guards
2. **A `choose` or `cond` block** ‚Äî dedicated multi-arm boolean dispatch
3. **Extend `match` to support literal patterns** ‚Äî `match` on integer ranges etc.

Until then, `explain` with named `when` entries serves this role. The `explain`
block remains dual-purpose: documentation prose AND structured branching.
