# C Runtime Bug Fixes — Round 2

**Date:** 2026-03-13
**Status:** Planned (not yet implemented)
**Scope:** `prove-py/src/prove/runtime/` — bugs and optimizations found in the second deep-analysis pass.
**Predecessor:** `02-runtime-bug-fixes.md` (archived; items there are done or deferred)

---

## Context

A second full audit of the C runtime found 5 new bugs (2 critical, 2 high, 1 medium) and
2 optimizations not covered by the first pass, plus one Python compiler bug in the emitter.

Design principle is unchanged: **the compiler is the primary safety mechanism.**
Runtime guards only earn their place for things the compiler cannot enforce — OOM, external
input, OS API contracts, and C internal logic errors.

## Bug Triage

### Critical

#### #1 — Refcount leaks in every `_deserialize_table` call
- **File:** `prove_store.c:172–196`
- **Problem:** `_read_str()` and `prove_string_from_cstr()` allocate strings with `refcount=1`.
  `prove_store_table_new` and `prove_store_table_add_variant` each call `prove_retain()` on the
  strings passed to them, bumping refcount to 2, but the caller's original reference is
  never released. Every `prove_store_table_inputs` call therefore leaks one refcount per
  column name, per variant name, and the table name itself. In a long-running store-backed
  program, these objects accumulate and never free.

  Affected variables:
  - `tname` (line 172): created by `prove_string_from_cstr`, retained by `table_new`, never released
  - `col_names[i]` (lines 164–169): created by `_read_str`, retained by `table_new`, array freed but strings not released
  - `vname` (line 187): created by `_read_str`, retained by `add_variant`, never released
  - `row[c]` (lines 190–191): created by `_read_str`, retained by `add_variant`, never released

- **Category:** Memory leak in C internal logic
- **Fix:** Call `prove_release()` on each local reference after the `table_new`/`add_variant`
  call has retained it:

  ```c
  // After: prove_store_table_new(tname, col_count, col_names)
  prove_release(tname);
  // After: col_names[i] = _read_str(f)  ... prove_store_table_new(...)
  for (int64_t i = 0; i < col_count; i++) prove_release(col_names[i]);
  // After: prove_store_table_add_variant(table, vname, row)
  prove_release(vname);
  for (int64_t c = 0; c < col_count; c++) prove_release(row[c]);
  ```

#### #2 — No NULL check on `calloc` in `prove_store_diff`; multiplication can overflow
- **File:** `prove_store.c:301–308`
- **Problem:** Three `calloc` calls allocate the diff arrays without NULL checks:

  ```c
  Prove_DiffVariant *added   = calloc(max_add, sizeof(Prove_DiffVariant));
  Prove_DiffVariant *removed = calloc(max_rem, sizeof(Prove_DiffVariant));
  Prove_DiffChange  *changed = calloc(max_chg > 0 ? max_chg : 1, sizeof(Prove_DiffChange));
  ```

  OOM here causes the next write through `added[nadd]` to segfault instead of panicking cleanly.

  Additionally, `max_chg = old_t->variant_count * old_t->column_count` multiplies two
  `int64_t` values. For large tables this can overflow to a negative number, which is then
  cast to `size_t`, producing a huge allocation that `calloc` fails — and then the missing
  NULL check causes a crash.

- **Category:** OOM handling, C undefined behavior (integer overflow)
- **Fix:**

  ```c
  // Guard the multiplication
  if (old_t->column_count > 0 &&
      old_t->variant_count > INT64_MAX / old_t->column_count) {
      prove_panic("store diff: table too large");
  }
  int64_t max_chg = old_t->variant_count * old_t->column_count;

  Prove_DiffVariant *added = calloc((size_t)max_add, sizeof(Prove_DiffVariant));
  if (!added) prove_panic("store diff: out of memory");
  Prove_DiffVariant *removed = calloc((size_t)max_rem, sizeof(Prove_DiffVariant));
  if (!removed) prove_panic("store diff: out of memory");
  Prove_DiffChange *changed = calloc((size_t)(max_chg > 0 ? max_chg : 1), sizeof(Prove_DiffChange));
  if (!changed) prove_panic("store diff: out of memory");
  ```

### High

#### #3 — OOB access in `prove_store_merge` addition-conflict loop
- **File:** `prove_store.c:510–516`
- **Problem:** When two sides both add the same variant (addition conflict), the merge loop
  iterates `base->column_count` to fill the conflict's value lists:

  ```c
  for (int64_t col = 0; col < base->column_count; col++) {
      if (local->added[l].values)
          prove_list_push(..., local->added[l].values[col]);   // ← OOB possible
  ```

  But `local->added[l].values` was allocated with `new_t->column_count` entries inside
  `prove_store_diff`. If a schema migration changed the column count between base and the
  local diff's source table, `base->column_count` can exceed `new_t->column_count`, causing
  an out-of-bounds read.

- **Category:** Logic error in C code
- **Fix:** Cap the loop at the minimum of base and diff column counts, or track the
  per-entry value count in `Prove_DiffVariant` (preferred). Quick fix:

  ```c
  int64_t ncols = base->column_count < local->added[l].value_count
                  ? base->column_count : local->added[l].value_count;
  for (int64_t col = 0; col < ncols; col++) {
  ```

  If `value_count` is not yet tracked in `Prove_DiffVariant`, add it in
  `prove_store_diff` when allocating the values array.

#### #4 — `prove_network_server_inputs` ignores the `host` parameter
- **File:** `prove_network.c:119–135`
- **Problem:** The function null-checks `host` but never uses it for binding:

  ```c
  Prove_Result prove_network_server_inputs(Prove_String *host, int64_t port) {
      if (!host) return _socket_error("bind");  // only null-check
      ...
      sa.sin_addr.s_addr = INADDR_ANY;          // host is silently ignored
  ```

  A caller passing `"127.0.0.1"` to restrict the server to loopback, or a specific
  interface address, gets a server that listens on all interfaces instead. This is a
  silent security/correctness issue.

- **Category:** Logic error — OS API misuse
- **Fix:** Resolve `host` using the existing `_resolve_and_fill` helper, which already
  handles IPv4 name resolution:

  ```c
  struct sockaddr_in sa;
  if (_resolve_and_fill(host, port, &sa) < 0) {
      CLOSE_SOCKET(fd);
      return _gai_error("bind");
  }
  // Remove the memset + manual sa construction that follows
  ```

  For `"0.0.0.0"` (bind-all) the resolver returns `INADDR_ANY` naturally. The existing
  client-connect path (`prove_network_socket_inputs`) already does this correctly.

### Medium

#### #5 — Error path in variant deserialization leaks already-read strings
- **File:** `prove_store.c:187–191`
- **Problem:** When reading a variant's column values fails partway through the loop, the
  already-read strings in `row[0..c-1]` are leaked — `free(row)` frees the array pointer
  but not the individual `Prove_String*` objects inside it. `vname` is also leaked:

  ```c
  Prove_String *vname = _read_str(f);
  if (!vname) { fclose(f); free(row); prove_release(table); return NULL; }
  for (int64_t c = 0; c < col_count; c++) {
      row[c] = _read_str(f);
      if (!row[c]) { fclose(f); free(row); prove_release(table); return NULL; }
      //           ↑ leaks vname and row[0..c-1]
  ```

- **Category:** Error-path memory leak in C code
- **Fix:**

  ```c
  if (!row[c]) {
      for (int64_t j = 0; j < c; j++) prove_release(row[j]);
      prove_release(vname);
      fclose(f); free(row); prove_release(table); return NULL;
  }
  ```

## Python Compiler Bug

#### #6 — Option→Boolean coercion only handles bare identifiers (`_emit_stmts.py:311–317`)
- **File:** `prove-py/src/prove/_emit_stmts.py:311–317`
- **Problem:** When a `validates` function's last expression is `Option<T>` and the return
  type is `Boolean`, the emitter is supposed to emit `.tag == 1` to check for Some.
  But the guard requires the expression to be a bare `IdentifierExpr`:

  ```python
  if isinstance(ret_type, PrimitiveType) and ret_type.name == "Boolean":
      if isinstance(expr_type, GenericInstance) and expr_type.base_name == "Option":
          if isinstance(expr, IdentifierExpr):        # ← misses call expressions
              emit_val = f"{expr.name}.tag == 1"
  ```

  When the last expression is a call that returns `Option<T>` (e.g. `Text.find(s, p)`),
  the tag check is skipped silently. The struct value from the call gets assigned directly
  to `bool`, which is either a C compiler warning or a wrong result (typically always
  `true` because a non-zero struct compares non-zero).

- **Category:** Compiler emitter bug — wrong C generated for `validates` functions
- **Fix:** Always materialise the Option expression into a temp and check `.tag`:

  ```python
  if isinstance(ret_type, PrimitiveType) and ret_type.name == "Boolean":
      if isinstance(expr_type, GenericInstance) and expr_type.base_name == "Option":
          # Must work for any expression, not just identifiers
          opt_tmp = self._tmp()
          opt_ct = map_type(expr_type)
          self._line(f"{opt_ct.decl} {opt_tmp} = {emit_val};")
          emit_val = f"({opt_tmp}.tag == 1)"
  ```

## Optimizations

#### O8 — Dead null guards in `prove_format.c` (not covered by `future/01`)
- **File:** `prove_format.c:5,21,36`
- `prove_format_pad_left`, `prove_format_pad_right`, and `prove_format_center` all open with
  `if (!s) s = prove_string_from_cstr("");` — dead code, `String` is never null in Prove.
  `future/01` lists `prove_text.c` and others but missed this file.
- **Fix:** Remove the three dead guards. Covered by the same sweep as future/01 Item 1.

#### O9 — `prove_array_set` always copies; COW fast-path not wired in
- **File:** `prove_array.c:47–57`
- `prove_array_set_mut` exists and is correct for in-place mutation, but `prove_array_set`
  never calls it — it always allocates a full copy regardless of refcount. The COW fast-path
  from `future/02 O3` is planned but the hook is not yet in place.
- **Fix:** Add at the top of `prove_array_set`:

  ```c
  if (arr->header.refcount == 1) {
      return prove_array_set_mut(arr, idx, val);
  }
  ```

  This is safe because the emitter only calls `prove_array_set` when the array may be
  aliased. When refcount is 1 there is exactly one owner and mutation is sound.
